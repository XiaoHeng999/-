# 硬性约束

每次会话你**必须且只能完成一个任务**。完成一个任务后：
1. 运行反馈检查
2. 更新 `{{PLANS_DIR}}/state.json` 状态为 `done`
3. 提交代码（包含 state.json 的状态更新）
4. 输出结果 — **禁止开始第二个任务**

如果你完成一个任务后发现还有其他未完成的任务，**忽略它们**。外层循环会在下一次迭代中用全新上下文处理。

在一次会话中处理多个任务是**严重错误**。完成一个就停下来。

# 任务来源

任务以 markdown 文件形式存放在 `{{PLANS_DIR}}/` 目录下，每个文件对应一个垂直切片。任务的状态和依赖关系通过以下两个文件管理：

- **`{{PLANS_DIR}}/state.json`**：记录每个任务的状态（`pending` / `doing` / `done`），当前状态会作为上下文注入
- **`{{PLANS_DIR}}/index.md`**：任务总览和依赖关系图

## 任务选择规则

1. 从注入的 state.json 中找到所有状态为 `pending` 的任务
2. 读取对应 issue 文件，检查 `Blocked by` 字段
3. 仅选择**所有依赖任务均已 `done`** 的 `pending` 任务
4. 如果有多个可执行任务，按编号顺序选择最小的
5. 开始执行前，将 `state.json` 中对应任务状态更新为 `doing`
6. 完成后更新为 `done`

如果所有任务均为 `done`，输出 `<promise>NO MORE TASKS</promise>`。

# 项目目录结构

项目代码和测试代码有明确的目录约定：

- **`src/`**：所有项目源代码必须放在此目录下，禁止在项目根目录创建业务代码文件
- **`tests/`**：所有测试代码必须放在此目录下
  - 单元测试：`tests/test_*.py`
  - 端到端测试：`tests/test_e2e/`

**禁止在项目根目录直接放置 `.py` 业务代码文件。** 根目录只保留配置文件（如 `pyproject.toml`）、文档和工具脚本。

# 实现方式

**你必须使用 `/tdd` 技能来完成所有实现。这是强制要求，不允许跳过。**

TDD 红-绿-重构循环：

## 红：写一个失败的测试

## 绿：写最小化的实现代码让测试通过

## 重构：清理代码，保持测试通过

## 红：再写下一个失败的测试

重复以上循环直到实现完成。

每次实现新功能或修复 bug 时，都必须通过 `/tdd` 技能启动，让技能引导你完成完整的红-绿-重构流程。

# 反馈检查

提交代码之前，必须依次运行以下反馈检查：

## 单元测试
```bash
uv run pytest tests/ -v --tb=short
```

## 端到端测试
```bash
uv run pytest tests/test_e2e/ -v --tb=short
```

## Lint 检查
```bash
uv run ruff check . --fix
```

## 构建验证
```bash
uv run python -c "import 主模块; print('构建验证通过')"
```

**所有检查必须全部通过。如果任一检查失败，必须修复后再提交。**

# CI 测试

项目使用 GitHub Actions 进行持续集成。每次推送到远程仓库后会自动运行 CI 测试。

提交并推送后，使用以下命令检查 CI 状态：

```bash
gh run list --limit 1
gh run watch
```

如果 CI 失败，必须查看失败日志并修复：

```bash
gh run view --log-failed
```

确保 CI 通过后再继续下一个任务。
